Commit to Kubernetes, without touching the pipeline.

A Jenkins delivery pipeline where every stage — build, scan, promote, verify, roll back — is derived from one declarative file in the application repository. Onboarding a new service or a new cluster is configuration, not code. Demonstrated end-to-end on two live Kubernetes clusters, with a gated production promotion and a proven automatic rollback.

01 / Architecture

One artifact, promoted — never rebuilt.

The image is built once per commit and moves unchanged through every environment. Only the runtime environment differs between clusters, so what passed verification in staging is bit-for-bit what reaches production.

Why one registry, two clusters. Rebuilding per environment means production runs an artifact nothing verified. Building once and promoting by tag makes "it passed in staging" a statement about the exact bytes now running in production.
02 / Pipeline stages

Every stage earns its place.

Each stage exists to stop a specific failure class as early as possible — and each has a defined behaviour when it does.

StagePurposeOn failure
CheckoutResolves commit SHA and the real branch name behind Jenkins' detached-HEAD checkoutAbort
Load configParses and validates deploy-config.yaml; resolves which environments this branch may reachAbort with a precise schema error
TestUnit tests in a throwaway container — the controller holds no language runtimesAbort before any image exists
Build imageBakes build number, commit, branch and version into the artifactAbort
Scan imageTrivy vulnerability scan at configured severityMark unstable, or fail if failOnFindings
Push imagePublishes the immutable tag; skipped entirely if the branch targets no environmentAbort
Deploy → stagingRenders manifests, applies, blocks on real rollout completionDiagnostics printed, automatic rollback
Smoke testProves the service answers over HTTP from inside the clusterRollback and fail
Deploy → prodHuman approval gate, then the identical imageRollback; staging unaffected

A green rollout only means containers started. Ready is not the same as working — the smoke test is what separates "Kubernetes accepted my manifest" from "the service responds".

03 / Adapting to new repos & clusters

Onboarding is two files. Neither is pipeline code.

The brief asked for a solution scalable and adaptable to different Git repositories and Kubernetes clusters. That rules out per-project Jenkinsfiles — logic copied into ten repositories is ten places to fix a bug. Here the logic lives once, in a Groovy shared library; a repository contributes only its identity.

The entire Jenkinsfile
@Library('finacplus-cicd@main') _

deliveryPipeline()

Three lines. No stages, no credentials, no cluster names — nothing to drift out of date.

deploy-config.yaml — excerpt
environments:
  - name: staging
    cluster: kind-staging
    namespace: demo
    credentialId: kubeconfig-staging
    branches: ["main"]
    autoDeploy: true

  - name: prod
    cluster: kind-prod
    credentialId: kubeconfig-prod
    autoDeploy: false
    approval:
      required: true
Adding a third cluster is a block of YAML. Onboarding a service is two files. Neither touches pipeline code — which is what makes the claim testable rather than aspirational.
04 / Security

CI holds exactly the power it needs. Verified, not asserted.

Each kubeconfig authenticates as a namespaced ServiceAccount bound to a Role — never a ClusterRole. The permission surface below is read from the live API server. 

$ kubectl auth can-i --list -n demo --as=system:serviceaccount:demo:jenkins-deployer
deployments.apps    [get list watch create update patch]
replicasets.apps    [get list watch]
services            [get list create update patch]
pods, pods/log      [get list watch]
events              [get list]
pods/portforward    [create]
CapabilitystagingprodRationale
patch deploymentsyesyesRequired to deploy
watch deploymentsyesyesrollout status opens a watch stream
create pods/portforwardyesyesNarrowest grant that permits smoke verification
create podsnonoWould let CI schedule arbitrary workloads
delete deploymentsnonoDeploying never requires deleting
get secretsnonoNo reason to read them
pods in kube-systemnonoNamespaced Role — a hard boundary

Defence in depth

  • Container runs as UID 10001 — read-only root filesystem, all capabilities dropped, runAsNonRoot enforced by the kubelet.
  • The app's ServiceAccount gets no API token (automountServiceAccountToken: false) — it never talks to Kubernetes, so it holds no credential.
  • Secrets never enter Groovy. The kubeconfig is dereferenced by the shell as $KUBECONFIG_FILE; interpolating a credential in Groovy is an exfiltration path Jenkins explicitly warns about.
  • Jenkins is configured as code. Credentials are injected from the environment at boot — the config file in Git contains references, never values.
  • Trivy scans every image before it can be pushed.
05 / Failure handling

Two fault classes. Two nets. Zero downtime.

A defect unit tests can catch never produces an image at all. A defect that only manifests once deployed — conditional on environment configuration injected at deploy time — passes CI and must be caught by the rollout itself. Both were demonstrated live.

The bad build cannot displace the good one

With maxUnavailable: 0, a new pod must pass readiness before any old pod retires:

The build fails, staging returns to the last good image, and production is never reached — the pipeline aborts before the approval gate. No human was required for any of it.

06 / Constraints, named

What a workstation forces — and what production does instead.

This runs on a single machine, so several choices are local-environment compromises rather than recommendations. Stating them precisely is part of the engineering.

HereWhyProduction

On the approval gate. It is a control, not a safety net. It confirms a human intended this promotion at this moment — the evidence that the release is safe comes from the tests, the scan, and the staging verification that already ran against the identical artifact.
